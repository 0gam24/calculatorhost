# docs/21 — 콘텐츠 운영 표준

사이트 오픈 후 지속적인 콘텐츠 추가·수정·관리 절차.

---

## 1. 콘텐츠 제작 12단계 절차

각 단계는 건너뛰지 않는다.

```
1. 키워드·검색 의도 분석
   ↓
2. 토픽 클러스터 내 위치 확정
   ↓
3. 경쟁사 SERP 및 AI 답변 분석
   ↓
4. 콘텐츠 브리프 작성
   ↓
5. 초안 작성
   ↓
6. 1차 자료·데이터 보강
   ↓
7. 전문가 검수 (해당 시)
   ↓
8. 편집·교정
   ↓
9. 메타·스키마·이미지
   ↓
10. 발행
   ↓
11. 내부 링크 보강
   ↓
12. 성과 추적
```

### 1-1. 키워드·검색 의도 분석

- 시드 키워드 → 관련 키워드 확장 (Google Keyword Planner, Ahrefs, SEMrush)
- 검색 의도 분류 (Informational / Navigational / Commercial / Transactional)
- 월간 검색량, 경쟁도, CPC 확인

### 1-2. 토픽 클러스터 내 위치

PROJECT.md §B-2의 토픽 권한 맵에서 위치 확인:

- 메인 토픽 → 허브 페이지로
- 서브 토픽 → 클러스터 페이지로
- 롱테일 → 세부 글로

### 1-3. 경쟁사 SERP·AI 답변 분석

- Google 검색 결과 상위 10개 확인
- 각 사이트의 길이, 헤딩 구조, 다루는 하위 주제
- People Also Ask, Related Searches 수집
- ChatGPT, Perplexity에 같은 키워드 입력 → 어떤 답변·인용을 하는지

### 1-4. 콘텐츠 브리프 (작성 전 필수)

```markdown
# 콘텐츠 브리프

## 타겟 키워드
- 주: <시드 키워드>
- 보조: <롱테일 1>, <롱테일 2>

## 검색 의도
<Informational/Commercial 등>

## 타겟 독자
<페르소나 + 그들이 알고 싶은 것>

## 글의 약속(Promise)
첫 문단에서 명시할 결론 1줄.

## 다뤄야 할 핵심 질문 (H2)
1. <질문 1>
2. <질문 2>
3. <질문 3>
4. <질문 4>
5. <질문 5>

## 차별화 포인트
- <자체 데이터/인터뷰/경험>
- <경쟁사 안 다룬 관점>

## 분량
<약 N자 — 경쟁사 평균 + 30%>

## 1차 자료
- <출처 1>
- <출처 2>

## 내부 링크 후보
- <허브 페이지>
- <관련 클러스터 글 3개>

## 외부 권위 링크 후보
- <정부/학술/공식 문서>
```

### 1-5. 초안 작성

`docs/12-geo-ai-citation.md` §2 청킹 표준 준수.

### 1-6. 1차 자료·데이터 보강

- 인용에는 출처 명시
- 자체 실험·인터뷰·사례
- 통계는 단위·시점 명시
- 표·도표·인포그래픽

### 1-7. 전문가 검수 (해당 시)

YMYL 분야 (의료·금융·법률):
- 자격 보유 전문가
- 검수 사실 명시 (이름·자격·일자)

### 1-8. 편집·교정

- 맞춤법
- 가독성 (문장 길이, 단락 길이)
- 톤앤매너 일관성
- AI 생성 흔적 제거 (어색한 표현, 일반론적 문장)

### 1-9. 메타·스키마·이미지

- title, description, canonical
- OG 이미지 (1200×630)
- 페이지 유형별 스키마 (`docs/10`)
- 본문 이미지 alt
- LCP 이미지 처리 (`docs/08`)

### 1-10. 발행

- 슬러그 확정 (영문, 6단어 이내)
- 카테고리·태그
- 발행일 명시
- Git 커밋·PR

### 1-11. 내부 링크 보강

새 글 발행 후:
- 허브 페이지에서 새 글로 링크
- 같은 클러스터 기존 글 2~3개에서 새 글로 링크
- 새 글에서도 관련 글 3~5개로 링크

### 1-12. 성과 추적

발행 후:
- 1주 후: Search Console 인덱싱 확인
- 4주 후: 검색 노출·클릭 데이터
- 12주 후: 순위·전환 분석
- AI 답변 엔진 인용 모니터링

## 2. 콘텐츠 품질 게이트

발행 전 다음 항목 모두 통과:

### 2-1. 독창성

- [ ] 표절 검사 통과 (Copyscape, Originality.ai)
- [ ] 자체 데이터·관점·경험 포함
- [ ] 경쟁사 콘텐츠 단순 재구성 아님

### 2-2. 충분성

- [ ] 검색 의도를 완전히 충족하는 깊이
- [ ] People Also Ask 질문 모두 다룸
- [ ] 후속 행동(다음에 해야 할 것) 안내

### 2-3. 정확성

- [ ] 모든 사실 주장에 출처
- [ ] 통계는 단위·시점·출처 명시
- [ ] 인용 링크 살아있음 (404 없음)
- [ ] 딥링크는 **작성 시점 fetch 로 200 확인한 URL 만** 사용 — 미확인 시 기관 홈 URL 유지 + 본문에 고시번호·법령명 텍스트 명기
- [ ] law.go.kr 은 `/법령/<법령명>` 영구링크 형식만 (nttId·bbsId 게시판형 쿼리 URL 금지)
- [ ] 기준일 인라인 "(YYYY년 N월 기준)" 은 글당 본문 1회 + 변동성 큰 핵심 수치 옆 선택 — 모든 수치 옆 의무화 금지 (G4 date 토큰 중복 추출로 match_rate 희석)
- [ ] 기관(연도) 인라인 표기는 R78 §15-3 형식과 통일 — `[근거: 국세청 고시 2026-12]` 또는 `(법제처, 2026.04)`

### 2-4. 최신성

- [ ] 최신 데이터·기준 반영
- [ ] 6~12개월마다 재검토 일정 등록

### 2-5. 인용 가능성

- [ ] 청킹 표준 준수 (`docs/12` §2)
- [ ] 첫 문단 결론
- [ ] 헤딩 자체 완결적
- [ ] 비교·통계 시각화 (표, 차트)

### 2-6. 카니발리제이션 대조 (수동 발행)

- [ ] 발행 전 동일 cluster 기존 글의 title·keywords·slug 대조
  - 예: `grep -il '핵심키워드' src/content/articles/*.mdx` 후 매칭 글의 cluster 확인
- [ ] 타겟 쿼리가 기존 글과 동일하면 **신규 발행 금지** → 기존 글 갱신으로 전환 (§4 dateModified 정책 준수)

### 2-7. 보일러플레이트 변주 풀 (신규 발행 글부터)

정형 패턴이 전 글에 복제되면 AI 양산 신호가 된다. 다음 변주 풀을 적용:

- **TL;DR 라벨 로테이션**: "핵심만 30초" 고정 금지 — "한눈에 핵심", "30초 브리핑", "바쁘면 이것만" 등 2~3종과 로테이션. 단 G8 S3 정규식 `^(결론|정리|요약|마무리|마치며)` 를 촉발하는 라벨은 금지.
- **리스트 개수 가변**: "5가지/5단계" 고정 해제 — 3~7 가변.
- **결론 패턴 해제**: "## 결론: + 정확히 5불릿" 고정 패턴 사용 금지.
- **em-dash 헤딩 비율**: "명사구 — 부제" 형 헤딩은 전체 H2 의 50% 이하.
- **description 말미 변주**: "…표와 함께 2026년 N월 기준으로 정리했습니다" 상투구는 의미를 유지하며 표현 다양화.
- **"본인" 과용 상한**: 한 문단 2회 이하 권장.
- **H2 개수 분산**: 7~12 범위에서 글마다 분산.

적용 범위: **신규 발행 글부터**. 기존 451편 소급 금지 — 소급은 AdSense 심사 종료 후 `scripts/audit/` dry-run + `/batch` 별도 트랙으로 진행.

### 2-8. faq frontmatter 규격 (파이프라인 강제값 그대로 전사 — 새 수치 발명 금지)

- [ ] `faq`: 3~5문항, 답변은 120~220자 완결형 (그대로 인용 가능한 단위)
- [ ] `aiCitationQuestions`: 5~8개 (파이프라인 검증기 하한 5)
- [ ] brief faq count: 3~7
- aiCitationQuestions ↔ faq ↔ H2 매핑은 **권장** (강제 아님)

## 3. AI 생성 콘텐츠 정책

### 3-1. 금지

- ❌ AI 단독 생성 후 무편집 발행
- ❌ "AI가 썼다"고 부인하면서 그대로 발행
- ❌ AI로 대량 양산하여 SEO 게이밍

### 3-2. 허용

- ✅ AI 보조 작성 + 사람의 사실 확인 + 1차 경험 추가
- ✅ AI로 초안 → 전문가 검수·편집
- ✅ AI로 아이디어·구조 잡기 → 사람이 본문 작성

### 3-3. 투명성

편집 정책 페이지에 AI 보조 사실 명시 권장:

```
본 사이트는 AI 도구를 보조 수단으로 사용합니다.
모든 글은 사람이 사실 확인·편집·승인한 후 발행됩니다.
```

### 3-4. 2026 스팸 업데이트 표적

Google은 다음을 표적:
- 무가치한 AI 양산 콘텐츠
- 사이트 신뢰도와 무관한 주제로 확장
- 스폰서 콘텐츠를 일반 콘텐츠로 위장
- 만료 도메인 활용 콘텐츠 도배

## 4. 콘텐츠 갱신 정책

### 4-1. 정기 재검토

- 모든 글: **12개월마다** 재검토 일정 등록
- YMYL 글: **6개월마다**
- 시사성 있는 글: 더 자주

### 4-2. dateModified 정책

- ✅ 본문 실질 변경 시에만 frontmatter `updatedAt` 갱신
- ❌ 오타 수정, CSS 변경 등으로 갱신 (스팸 분류 위험)
- ❌ 본문 무변경 재검토 시 `lastReviewed` 단독 갱신 — 재검토는 반드시 `dataValidAsOf` 갱신 + 출처 링크 생존·수치 유효 확인을 동반할 때만 `lastReviewed` 기록
- 재검토 주기는 YMYL 6개월 (§4-1) 과 연동 — 그보다 잦은 갱신 금지
- 법정 수치가 연 단위로 개정되면 **본문 정정 + `updatedAt` 세트** 의무 (날짜만 갱신 금지)
- ❌ 기존 451편 소급 일괄 backfill 명시적 금지 — 근거: R94-8 사고 (LLM 이 임의 updatedAt 을 박아 전수 제거)

### 4-3. 노후 콘텐츠 처리

- 여전히 유효 → 갱신 후 dateModified
- 부분적 변경 필요 → 인플레이스 수정
- 완전 무용 → 301 리다이렉트 또는 noindex
- 가치 있지만 시점 지난 글 → "2024년 기준"으로 명시 후 보존
