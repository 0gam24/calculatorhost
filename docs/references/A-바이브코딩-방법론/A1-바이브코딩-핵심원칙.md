# A1 — 바이브코딩 핵심 원칙

> **참조 시점**: 새 프로젝트 시작 전 · AI 작업 중 막힐 때 · 운영 체계 점검 시

---

## 1. 3-레이어 운영 OS

AI와 협업하는 모든 작업은 세 개의 레이어로 관리한다.

```
┌─────────────────────────────────────────┐
│ [외부 레이어]  다중 프로젝트 운영          │
│ → 노트-First / 시간 예산 / 대시보드       │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│ [실행 레이어]  단일 프로젝트 라이프사이클    │
│ → PRD → 설계 → 구현 → 테스트 → 배포      │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│ [제어 레이어]  AI 행동 자동 제어            │
│ → AGENTS.md + Hook + 멀티 에이전트       │
└─────────────────────────────────────────┘
```

**한 줄 요약**: 외부에서 "어디로", 실행에서 "어떻게", 제어에서 "AI가 길을 벗어나지 않게".

## 2. 안티패턴 — 절대 하지 말 것

### 운영 안티패턴
| 패턴 | 위험 |
|---|---|
| "Accept All" 무한 루프 | AI 결과 검토 없이 모두 승인 → 통제 불능 |
| 노트 없이 코드부터 시작 | 의사결정 근거가 휘발 |
| 시간 예산 없이 작업 | "조금만 더..."가 1시간으로 부풀음 |
| 도구 결정에 말려 시작 못 함 | 완벽한 도구 선택보다 실행이 우선 |

### 구현 안티패턴
| 패턴 | 위험 |
|---|---|
| "전체 서비스를 한 번에 만들어줘" | 범위 너무 넓으면 AI도 헤매고 오류 폭증 |
| Git 미사용 | "이전 상태로 되돌려줘" 불가능 |
| 코드 먼저, 설계 나중 | DB 구조 나중 변경 = 절반 재작성 |
| 폴더 구조 없이 시작 | 수십 파일 동시 수정 → 스파게티 |
| 같은 실수 반복 → 룰 파일에 반영 안 함 | 매번 같은 교정 반복 |
| 거대한 단일 파일 (500줄+) | AI도 사람도 헤맴 |

### 제어 안티패턴
| 패턴 | 위험 |
|---|---|
| 프롬프트만으로 AI 제어 | 지속성·일관성·강제성 없음 |
| 코드 작성과 리뷰를 같은 에이전트가 | 자기 편향 발생 |
| 규칙에 이유 안 남김 | 시간 지나면 무시되거나 잘못 제거됨 |
| 수십 규칙 정리 안 함 | 하네스 비대·규칙 충돌 |

## 3. 핵심 패턴

### 3-1. 노트-First 원칙 ★

코드 작성 전 4가지가 노트에 답해져 있어야 함:
1. **무엇을** 만드는가? (목적)
2. **왜** 만드는가? (이유)
3. **어떻게** 만들 것인가? (전략)
4. **어떻게 검증**할 것인가? (성공 기준)

### 3-2. 성공적인 바이브 코딩 5단계 ★★★

1. **기획 + PRD** — 제품 요구사항 명세서. AI가 참조할 청사진
2. **설계 + 아키텍처** — 데이터 모델, API, 시스템 구성
3. **AI 협업 구현** — 모듈 분할 + SDD(Spec-Driven Development)
4. **지속적 테스트** — 브라우저 에이전트가 사람처럼 QA 수행
5. **배포 + 유지보수** — AI가 모니터링·피드백 분석·개선 제안

### 3-3. 모듈 분리 1줄 기준

> **"이 기능만 따로 테스트할 수 있는가?"**

독립적으로 동작하고 검증 가능하면 좋은 모듈.

### 3-4. 컨텍스트 엔지니어링 ★

> **"LLM이 CPU라면, 컨텍스트 윈도우는 RAM이다."** — Karpathy

- 작업 시작 시 AI에게 **어떤 파일을 먼저 읽게 할지** 결정
- 큰 작업은 컨텍스트 분할 (탐색 → 구현 분리)
- 불필요한 정보는 컨텍스트에 넣지 않기

### 3-5. SDD (Spec-Driven Development)

명세를 먼저 작성하고, 그 명세를 기반으로 설계·코드·테스트 작성/검증.
큰 기능 추가 전 1~2 페이지 명세가 10시간 절약.

### 3-6. Git 원자 관리

- **원자적 커밋**: 하나의 기능 = 하나의 커밋
- **브랜치 전략**: 새 모듈은 별도 브랜치, 성공하면 merge / 실패하면 삭제
- **체크포인트**: 큰 변경 전 항상 git commit으로 안전 포인트

## 4. 하네스 엔지니어링

> **Agent = Model + Harness**

AI의 행동을 결정하는 시스템 설계. AGENTS.md는 "AI의 행동을 결정하는 코드"다.

### 4-1. 4가지 신호 체계

1. **지침** (코딩 컨벤션, 기술 스택 규칙) → AGENTS.md
2. **제약** (파일 수정 전 자동 검사) → Hook
3. **자동화** (반복 워크플로우 표준 절차) → Skills
4. **도구 접근** (외부 서비스 연결) → MCP

### 4-2. 5계층 설정 분리

| 파일 | 범위 | Git |
|---|---|---|
| `AGENTS.md` | 프로젝트 전체 | 커밋 |
| `[디렉토리]/AGENTS.md` | 디렉토리 | 커밋 |
| `.claude/settings.json` | 프로젝트 | 커밋 |
| `.claude/settings.local.json` | 개인 | **.gitignore 필수** |
| `~/.claude/CLAUDE.md` | 사용자 전체 | 별도 |

### 4-3. Hook 3종

- **PostToolUse**: 파일 수정 직후 (예: prettier 자동 포맷)
- **SubagentStop**: sub-agent 작업 종료 시 (예: lint 검증)
- **Stop**: 전체 세션 종료 시 (예: 빌드 검증)

### 4-4. 규칙에 이유 주석

```markdown
## 금지 사항
<!-- Cloudflare Pages는 Node.js API를 지원하지 않으므로 -->
- fs, path 등 Node.js 내장 모듈을 클라이언트 코드에서 사용하지 않는다
```

## 5. 멀티 에이전트 패턴

### 5-1. 5가지 패턴

1. **작성 vs 리뷰 분리** — 자기 편향 방지
2. **프론트엔드 vs 백엔드** — 디렉토리별 AGENTS.md
3. **탐색 vs 구현 분리** — 컨텍스트 절약 (가장 자주 활용)
4. **사이트별 병렬** — Agent Teams로 터미널 병렬
5. **Sub-agent 호출 4요소**: 역할 / 범위 / 제약 / 출력 형식

### 5-2. Claude Code vs 자율 에이전트

| 역할 | Claude Code | 자율 에이전트 |
|---|---|---|
| 신규 기능 설계 | **강함** | 보통 |
| 반복 유지보수 | 세션마다 재시작 | **메모리로 누적** |
| 24/7 자율 실행 | 불가 | **크론, 메시지** |

→ **신규/창의 = Claude Code / 반복/지속 = cron + sub-agent**

### 5-3. 보안 체크리스트

- ✅ API 키를 에이전트 간 **공유하지 않음**
- ✅ 각 에이전트에 **최소 권한만** 부여
- ✅ 인터넷 접근은 **default-deny + allowlist**
- ✅ memory/workspace **주기적 백업**
