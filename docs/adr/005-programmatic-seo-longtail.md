# ADR-005: 프로그래매틱 SEO를 Phase 2 최우선 확장 모델로 채택

**상태**: 수락됨  
**일자**: 2026-04-24

## 맥락
2026년 AdSense 수익 아이템 전략 보고서는 **프로그래매틱 SEO(Programmatic SEO)** 를 가장 각광받는 수익 모델로 지목. 방대한 산업 데이터를 API 연동해 수만 개 롱테일 랜딩 페이지를 자동 생성하는 방식. 페이지당 트래픽은 미미해도 10만 페이지 누적 시 오가닉 트래픽 폭발적. 콘텐츠 제작 한계 비용이 0에 수렴.

## 결정
**MVP (15개 계산기) 안정화 직후 Phase 2의 최우선 확장 작업으로 프로그래매틱 SEO 페이지 양산 채택.**

### 1차 대상 (공공데이터 API 기반)
1. **아파트 단지별 실거래가 페이지** — `/시세/[아파트ID]`
   - 국토부 실거래가 API 연동
   - 전국 공동주택 약 3-5만 단지 커버 예상
   - 각 페이지: 단지 개요 + 6개월 실거래가 + 양도세/취득세 자동 계산 + 전월세 전환 분석
2. **지역별 공시지가 페이지** — `/공시지가/[행정동]`
   - 행안부 법정동코드 + 국토부 공시지가 API
   - 전국 약 1.7만 읍면동
3. **은행별 대출 상품 페이지** — `/대출상품/[상품ID]`
   - 금감원 금융상품 공시 API
   - 약 500-1000개 상품

→ **예상 총 페이지 수: 5-7만+**

## 근거
1. 보고서 §4.3: "콘텐츠 제작 한계 비용 0 + 무한 확장성 + 애드센스와 최상 시너지"
2. 우리 공공데이터 API 카탈로그에 필요한 데이터 소스 이미 식별됨 (api-catalog.md)
3. 롱테일 키워드 ("서울 강남구 은마아파트 실거래가") 포획 — 2-3티어 키워드와 시너지
4. knowingasset.com도 유사 구조 사용 중 (`/app/land/house/[ID]`)

## 대안 고려
- **블로그 가이드 대량 작성**: 수작업 한계. AdSense 최저 공개 기준 위반 위험
- **AI 대량 생성**: Google 스팸 정책 위반 위험
- **사용자 UGC**: 초기 견인력 부족

## 실행 전제
- MVP 15개 계산기 안정 궤도 (월 1만 세션)
- Cloudflare Pages 빌드 시간 상한(20분) 고려 → 점진적 ISG/ISR 검토
- API 호출 쿼터 관리 (하루 1회 배치 빌드)

## 리스크
- 페이지 품질 낮으면 YMYL 정책 위반 (얇은 콘텐츠)
  → 대응: 각 페이지에 **고유 분석 값**(평균 대비 %, 주변 단지 비교) 포함
- Cloudflare Pages 빌드 시간 폭증 → 점진적 생성 전략 (신규는 ISR)
- API 스키마 변경 시 전체 페이지 영향 → zod 스키마 + 모니터링
